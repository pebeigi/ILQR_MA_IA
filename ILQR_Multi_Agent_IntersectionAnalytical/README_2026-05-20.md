# ILQR Multi-Agent Vehicle Formulation

Date: 2026-05-20

This note records the current formulation in the repository after the smoothing
and terminal-arrival updates.  It is intended as a compact technical reference
before further warm-start changes.

## State, Control, and Horizon

Each agent has a 4D state

```text
x_i = [px_i, py_i, v_i, theta_i]
```

where `px, py` are position, `v` is speed, and `theta` is heading.  Each agent
has a 2D control

```text
u_i = [kappa_i, a_i]
```

where `kappa` is path curvature and `a` is longitudinal acceleration.

The full multi-agent state stacks all agents:

```text
x = [x_0, x_1, ..., x_{N-1}]
```

Current experiment constants in `main.py`:

```text
dt = 0.1 s
horizon_steps = 200
nominal horizon = 20.0 s
state_dim_per_agent = 4
control_dim_per_agent = 2
stop_radius = 2.0 m
pass_through_radius = 1.0 m
stop_after_arrival_steps = 5
max_iterations = 300
alpha_scaling = 0.5
convergence_tol = 1e-3
lq_solver_type = open_loop
```

## Continuous Dynamics

For each vehicle:

```text
px_dot      = v cos(theta)
py_dot      = v sin(theta)
v_dot       = a
theta_dot   = v kappa
```

The concatenated dynamics evaluate each agent independently:

```text
x_dot = f(t, x, u_0, ..., u_{N-1})
```

The rollout in the main experiment uses explicit Euler integration:

```text
x_{k+1} = x_k + dt f(t_k, x_k, u_k)
```

The RK4 integrator still exists in `ilq/rollout.py`, but `main.py` currently
passes `use_euler=True` to the solver.

## Active Cost Stack in `main.py`

For each player/agent `i`, `build_multi_agent_game()` adds these costs:

```text
1. AgentRunningSpeedCost
2. AgentRunningDestinationCost
3. AgentArrivalHoldCost
4. BatchedStaticObstacleRepulsionCost
5. PairwiseAgentRepulsionCost for each j != i
6. AgentProximitySpeedCost for cross-traffic pairs with positive weight
7. AgentYCeilingCost when enabled
8. AgentXCeilingCost when enabled
9. AgentLaneKeepingCost when enabled
10. AgentSpeedDependentControlCost
11. AgentFullTerminalCost at k = horizon_steps - 1
```

The cost container sums all terms for a player.  It also adds tiny regularizers:

```text
state_regularization   = 1e-6
control_regularization = 1e-6
```

## Local Quadratic Approximation Convention

Each cost term returns a `QuadraticApprox`:

```text
cost approx =
  0.5 x^T Q x + q^T x + const
  + sum_j (0.5 u_j^T R_j u_j + r_j^T u_j)
```

The LQ game solver works in local coordinates around the nominal trajectory:

```text
dx = x - x_ref
du_j = u_j - u_ref_j
```

The packed local terms are gradients at the nominal point:

```text
l_i[k]    = Q_i[k] x_ref[k] + q_i[k]
rho_ij[k] = R_ij[k] u_ref_j[k] + r_ij[k]
```

Most custom nonlinear costs compute the exact local gradient, choose a positive
semidefinite state/control curvature for numerical stability, then set

```text
q     = grad_x - Q x0
const = f0 - grad_x^T x0 + 0.5 x0^T Q x0
```

For control-only terms:

```text
r = grad_u - R u0
```

The `QuadraticApprox.S` cross-term field exists in the container, but
`pack_for_lq_game()` currently does not pass state-control cross terms to the LQ
solvers.  For that reason, terms such as speed-dependent control cost and
position-dependent speed target keep exact first-order information and use PSD
diagonal/block curvature only.

## Smoothing Functions

### C2 Smooth Floor

Used instead of hard `max(value, floor)` in speed-floor terms.

Let

```text
offset = value - floor
width > 0
s = offset / width
```

Then

```text
smooth_floor(value, floor, width) =
  floor,                         offset <= 0
  value,                         offset >= width
  floor + width p(s),            otherwise
```

with

```text
p(s)  = 3 s^5 - 8 s^4 + 6 s^3
p'(s) = 15 s^4 - 32 s^3 + 18 s^2
p'' contribution = (60 s^3 - 96 s^2 + 36 s) / width
```

This is C2, exact below the floor, exact above the transition, and avoids the
kink of a hard clamp.

### C2 Smootherstep Gate

Used for signed arrival speed taper and arrival hold activation.

```text
smootherstep(s) =
  0,                              s <= 0
  1,                              s >= 1
  6 s^5 - 15 s^4 + 10 s^3,        otherwise
```

Derivative inside `(0, 1)`:

```text
30 s^4 - 60 s^3 + 30 s^2
```

### Stable Sigmoid

Used for pairwise moving/arrival gates:

```text
sigmoid(z) = 1 / (1 + exp(-z))
```

implemented in a numerically stable branch for positive and negative `z`.

### C2 Quadratic Hinge

Used for one-sided barrier/ceiling costs instead of
`0.5 * w * max(0, err)^2`.

For transition width `delta > 0`:

```text
f(err) = 0,                                      err <= 0
f(err) = 0.5 w err^2,                            err >= delta
f(err) = w delta^2 (1.5 s^3 - 1.5 s^4 + 0.5 s^5), 0 < err < delta
s = err / delta
```

Gradient in the transition:

```text
f'(err) = w delta (4.5 s^2 - 6 s^3 + 2.5 s^4)
```

Hessian in the transition:

```text
f''(err) = w (9 s - 18 s^2 + 10 s^3)
```

This is C2 and matches the exact quadratic value, gradient, and Hessian at
`err = delta`.

### Smooth Heading Error

Terminal heading cost uses the periodic proxy

```text
e_theta = 2 sin((theta - theta_desired) / 2)
```

The heading part of the cost is

```text
0.5 b_heading e_theta^2
= b_heading (1 - cos(theta - theta_desired))
```

Gradient:

```text
b_heading sin(theta - theta_desired)
```

Hessian:

```text
b_heading cos(theta - theta_desired)
```

The Hessian used in the LQ approximation is clipped to be PSD:

```text
max(0, b_heading cos(delta_theta))
```

## Active Cost Formulations

### AgentRunningSpeedCost

Purpose: track desired speed far from the destination, taper target speed to
zero near the terminal plane, and keep it zero after pass-through.

If a terminal heading is available:

```text
d_hat = [cos(theta_terminal), sin(theta_terminal)]
remaining = (destination - position)^T d_hat
s = (remaining - pass_through_radius) / arrival_speed_transition
gate = smootherstep(s)
v_target = v_desired gate
```

If no terminal heading is available, the gate can fall back to Euclidean
distance.  The active experiment uses terminal headings.

Cost:

```text
0.5 q_speed (v - v_target)^2
```

Quadraticization:

```text
err = v - v_target
grad_v = q_speed err
grad_pos = -q_speed err grad_pos(v_target)
Q[v,v] = q_speed
```

The exact local gradient is retained.  Position-speed cross curvature is not
sent to the LQ solver, so only PSD speed curvature is used.

### AgentRunningDestinationCost

Weak running spring to the destination:

```text
0.5 w_destination ||p_i - p_goal_i||^2
```

Quadraticization:

```text
Q[p,p] = w_destination I_2
q[p]   = -w_destination p_goal
const  = 0.5 w_destination p_goal^T p_goal
```

### AgentArrivalHoldCost

Purpose: once the agent reaches the terminal plane, keep it parked near the
destination inside the optimization rather than only freezing plots/CSV after
the solve.

Signed remaining distance:

```text
d_hat = [cos(theta_terminal), sin(theta_terminal)]
remaining = (destination - position)^T d_hat
s = (remaining - pass_through_radius) / arrival_hold_transition
before_gate = smootherstep(s)
hold_gate = 1 - before_gate
```

Cost:

```text
hold_gate * (
  0.5 w_hold_pos ||p_i - p_goal_i||^2
  + 0.5 w_hold_speed v_i^2
)
```

Quadraticization uses exact local gradient:

```text
base = 0.5 w_pos ||p - p_goal||^2 + 0.5 w_speed v^2
grad_p = hold_gate w_pos (p - p_goal) + base grad_p(hold_gate)
grad_v = hold_gate w_speed v
```

PSD curvature:

```text
Q[p,p] = hold_gate w_pos I_2
Q[v,v] = hold_gate w_speed
```

Then `q` and `const` are chosen to match the local value and gradient at the
nominal point.

### BatchedStaticObstacleRepulsionCost

For static obstacles `o_m`:

```text
sum_m w_obs / (||p_i - o_m||^2 + epsilon_obs)
```

Let

```text
delta_m = p_i - o_m
z_m = delta_m^T delta_m + epsilon_obs
```

Exact gradient:

```text
grad_p = sum_m -2 w_obs delta_m / z_m^2
```

PSD curvature approximation:

```text
H = (sum_m max(1e-8, 2 w_obs / z_m^2)) I_2
Q[p,p] = H
```

### PairwiseAgentRepulsionCost

For player `i` repelled by another agent `j`:

```text
w_pair gate_i gate_j / (||p_i - p_j||^2 + epsilon_pair)
```

When terminal headings are supplied, each moving gate uses signed remaining
distance:

```text
remaining_i = (destination_i - p_i)^T d_hat_i
arg_i = (remaining_i - stop_radius) / activation_width
gate_i = sigmoid(arg_i)
```

Gradient of the gate:

```text
grad_p_i gate_i = -(gate_i (1 - gate_i) / activation_width) d_hat_i
```

If terminal headings are absent, the fallback gate uses squared Euclidean
distance to destination.

Let

```text
delta = p_i - p_j
z = delta^T delta + epsilon_pair
g = gate_i gate_j
f0 = w_pair g / z
```

Exact local gradients:

```text
grad_i = w_pair (gate_j grad_gate_i / z - 2 g delta / z^2)
grad_j = w_pair (gate_i grad_gate_j / z + 2 g delta / z^2)
```

PSD relative-position curvature:

```text
H = max(1e-8, 2 w_pair g / z^2) I_2
Q_ii += H
Q_jj += H
Q_ij += -H
Q_ji += -H
```

### AgentProximitySpeedCost

Used for cross-traffic yielding.  It penalizes positive speed when another
agent is nearby:

```text
w_prox smooth_floor(v_i, 0, speed_transition_width) /
(||p_i - p_j||^2 + epsilon_prox)
```

Let

```text
vplus = smooth_floor(v_i, 0, width)
z = ||p_i - p_j||^2 + epsilon_prox
```

Exact gradient:

```text
grad_v_i = w_prox vplus' / z
grad_p_i = -2 w_prox vplus (p_i - p_j) / z^2
grad_p_j =  2 w_prox vplus (p_i - p_j) / z^2
```

PSD curvature:

```text
H_pos = max(1e-8, 2 w_prox vplus / z^2) I_2
Q[v_i,v_i] = max(0, w_prox vplus'' / z)
```

The position blocks use the same relative-position structure as pairwise
repulsion.

### AgentYCeilingCost

Soft ceiling on `py`:

```text
err = py_i - y_max
cost = C2_quadratic_hinge(err, weight, transition_width)
```

Quadraticization:

```text
grad_y = f'(err)
Q[y,y] = f''(err)
q[y] = grad_y - Q[y,y] y0
```

### AgentXCeilingCost

Soft ceiling on `px`:

```text
err = px_i - x_max
cost = C2_quadratic_hinge(err, weight, transition_width)
```

Quadraticization is identical to the y-ceiling with the x index.

### AgentLaneKeepingCost

Optional lane-center quadratic:

```text
0.5 w_lane (px_i - lane_x)^2
```

and/or

```text
0.5 w_lane (py_i - lane_y)^2
```

For an x-lane term:

```text
Q[x,x] = w_lane
q[x] = -w_lane lane_x
const += 0.5 w_lane lane_x^2
```

### AgentSpeedDependentControlCost

Notebook-style state-dependent control cost:

```text
0.5 beta_2 kappa_i^2 v_floor_i^4 + 0.5 beta_3 a_i^2
```

where

```text
v_floor_i = smooth_floor(v_i, v_min, floor_width)
```

The LQ approximation currently contributes only a control Hessian evaluated at
the nominal speed:

```text
R_i = diag(beta_2 v_floor_i^4 + 1e-3, beta_3 + 1e-3)
r_i = [0, 0]
Q = 0
q = 0
```

The exact state-control cross terms of this nonlinear cost are not used because
`S` is not packed into the LQ game solver.

### AgentFullTerminalCost

Active only at

```text
k = horizon_steps - 1
```

Cost:

```text
0.5 b_px (px - px_goal)^2
+ 0.5 b_py (py - py_goal)^2
+ 0.5 b_speed (v - v_goal)^2
+ 0.5 b_heading (2 sin((theta - theta_goal)/2))^2
```

Current `main.py` uses `v_goal = 0`.

Quadraticization:

```text
Q[px,px] = b_px
Q[py,py] = b_py
Q[v,v]   = b_speed
grad_theta = b_heading sin(theta - theta_goal)
hess_theta = b_heading cos(theta - theta_goal)
Q[theta,theta] = max(0, hess_theta)
```

The affine terms are chosen so that `Q x + q` equals the true local gradient at
the nominal point.

## Linearization

Continuous-time dynamics are linearized by central finite differences around
the nominal point:

```text
xdot = f(t, x, u_0, ..., u_{N-1})

A[:,j] = (f(t, x + eps e_j, u) - f(t, x - eps e_j, u)) / (2 eps)

B_i[:,j] =
  (f(t, x, ..., u_i + eps e_j, ...) -
   f(t, x, ..., u_i - eps e_j, ...)) / (2 eps)
```

Default finite-difference step:

```text
eps = 1e-6
```

The continuous linearization is discretized with forward Euler:

```text
A_d = I + dt A
B_{d,i} = dt B_i
```

Thus the local LQ dynamics are

```text
dx_{k+1} = A_d[k] dx_k + sum_i B_{d,i}[k] du_i[k]
```

## ILQ Iteration

At each ILQ iteration:

1. Roll out the current strategies from `x0`.
2. Evaluate and store each player cost per step.
3. Check convergence using maximum state trajectory difference from the previous
   operating point.
4. Linearize the dynamics along the nominal trajectory.
5. Quadraticize and pack all player costs along the nominal trajectory.
6. Solve the local finite-horizon LQ dynamic game.
7. Repeat until convergence or maximum iterations.

The rollout strategy is:

```text
u_i[k] =
  u_ref_i[k]
  - P_i[k] (x_k - x_ref[k])
  - alpha_scaling alpha_i[k]
```

Control safety clamps:

```text
kappa in [-0.6, 0.6]
a     in [-8.0, 8.0]
```

## Open-Loop LQ Game Solver

The active solver is `open_loop`.

The local LQ game at step `k` uses the packed local cost terms:

```text
0.5 dx^T Q_i[k] dx + l_i[k]^T dx
+ sum_j (0.5 du_j^T R_ij[k] du_j + r_ij[k]^T du_j)
```

The open-loop solver works backward.  It keeps value functions

```text
V_i(dx) = 0.5 dx^T Z_i dx + zeta_i^T dx + const
```

At each step, it forms the coupled stationarity matrix:

```text
S_ii = R_ii + B_i^T Z_i B_i
S_ij = B_i^T Z_i B_j,  i != j
```

Feedback-like gains for the local linearized game are solved by least squares:

```text
S P = concat_i(B_i^T Z_i A)
```

Affine terms are solved from:

```text
S alpha = concat_i(B_i^T zeta_i + r_ii)
```

The closed-loop local dynamics used in the value recursion are

```text
F = A - sum_i B_i P_i
beta = -sum_i B_i alpha_i
```

Value recursion:

```text
Z_i <- F^T Z_i F + Q_i + sum_j P_j^T R_ij P_j

zeta_i <- F^T (zeta_i + Z_i beta)
          + l_i
          + sum_j P_j^T (R_ij alpha_j)
          - sum_j P_j^T r_ij
```

The returned strategy has

```text
du_i = -P_i dx - alpha_i
```

During rollout, the feedforward term is scaled by `alpha_scaling`.

## Feedback LQ Game Solver

The feedback solver is available but not active in the current experiment.  It
also returns

```text
du_i[k] = -P_i[k] dx[k] - alpha_i[k]
```

It initializes terminal value functions from the final quadraticized state cost:

```text
Z_i[-1] = Q_i[-1]
zeta_i[-1] = l_i[-1]
```

At each backward step it forms the same coupled Nash stationarity structure,
but solves gains and feedforward terms in one block system:

```text
S X = Y
```

where the first `xdim` columns of `X` are gains and the final column is the
affine term.  The solver applies Gershgorin-style diagonal regularization to
the coupled matrix when needed:

```text
diagonal increased until each Gershgorin lower bound >= 1e-3
```

## Terminal and Arrival Handling

There are two distinct terminal concepts:

1. Solver horizon terminal cost at `k = horizon_steps - 1`.
2. Per-agent pass-through arrival, detected when distance to destination is less
   than `pass_through_radius`.

The old behavior only froze agents after `solver.solve()` for output.  The
current formulation also includes arrival behavior inside the optimization:

```text
AgentRunningSpeedCost: target speed tapers to zero near destination.
AgentArrivalHoldCost: post-arrival running hold cost keeps agent parked.
PairwiseAgentRepulsionCost: pairwise gate deactivates near/past terminal plane.
AgentFullTerminalCost: soft final anchor at the end of the horizon.
```

The post-solve `freeze_arrived_agents()` still exists for clean plots and CSVs,
but it should now be a presentation safeguard rather than the only mechanism
preventing drift after arrival.

## Warm-Start Arrival Handling

The initial nominal trajectory is a hand-built feasible guess from
`make_initial_nominal_trajectory()`.  It is not the final optimized trajectory,
but it now uses the same terminal-direction idea as the costs while remaining
dynamically feasible.  The warm start is produced by integrating the vehicle
dynamics under braking/steering controls; it no longer teleports or manually
freezes the state at the destination.

For each agent, the warm start computes:

```text
d_hat = [cos(theta_terminal), sin(theta_terminal)]
delta = destination - position
remaining = delta^T d_hat
lateral_error = ||delta - remaining d_hat||
```

Near the destination, the target speed is limited by a stopping-distance rule:

```text
v_stop = sqrt(2 a_brake max(remaining - stop_buffer, 0))
```

where the current warm-start values are:

```text
max_warm_start_brake = 4.0
stop_buffer = 0.25
```

The braking limit is applied once the lateral error is small enough to be in the
terminal corridor.  Otherwise the warm start keeps moving toward the geometric
destination to avoid stopping beside the target.

The nominal acceleration is a clipped proportional speed correction:

```text
a = clip(2 (v_target - v), -max_warm_start_brake, 1.5)
```

Controls are then integrated normally through the same vehicle dynamics as the
solver rollout.  This keeps the magenta initial-guess curve close to the
destination without creating a nonphysical jump that the first ILQR rollout
cannot reproduce.
